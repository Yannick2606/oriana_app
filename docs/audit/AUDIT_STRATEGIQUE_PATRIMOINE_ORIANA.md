# Audit stratégique du patrimoine orIAna

**Version : 1.0**  
**Statut : Base consolidée à confronter et enrichir**  
**Nature : Audit de couverture stratégique, documentaire, fonctionnelle et cognitive**  
**Principe directeur : La Vision guide le développement. Le développement ne guide jamais la Vision.**

---

## 0. Objet du document

Ce document rassemble, structure et sécurise les idées majeures construites autour d'orIAna afin d'éviter leur perte au fil du développement.

Il ne remplace ni la Vision, ni la Constitution, ni la Méthode BORÉAL, ni les spécifications. Il sert à :

1. inventorier le patrimoine intellectuel du projet ;
2. confronter les idées à la documentation existante ;
3. détecter les oublis, doublons, contradictions et ambiguïtés ;
4. identifier le document existant à enrichir ;
5. relier la Vision aux capacités, fonctionnalités, composants, données, agents, tests et code ;
6. conserver les idées non encore planifiées sans les transformer prématurément en exigences ;
7. préparer les arbitrages stratégiques et les futures versions du produit.

Ce document constitue une **base de consolidation**. Il doit être confronté aux autres conversations, notes, fichiers, décisions et versions historiques du projet.

# 1. Identité et ambition d'orIAna

## 1.1 Raison d'être

orIAna vise à transformer l'expérience quotidienne d'une organisation en intelligence collective durable.

La plateforme ne doit pas être réduite à un CRM enrichi par l'intelligence artificielle. Elle a vocation à devenir une **plateforme cognitive d'organisation augmentée**, capable de :

- comprendre les situations ;
- structurer les données et les informations ;
- préserver les connaissances ;
- expliquer les raisonnements et décisions ;
- apprendre des réussites et des erreurs ;
- transmettre les savoirs ;
- développer les compétences individuelles et collectives ;
- améliorer durablement la performance ;
- accompagner la maturité de l'organisation.

## 1.2 Promesse stratégique

> Transformer les données, expériences, décisions, réussites et erreurs d'une organisation en connaissances explicables, en compétences réutilisables et en performance durable.

## 1.3 Invariants à préserver

- L'humain reste responsable de la décision finale.
- L'IA augmente l'humain ; elle ne se substitue pas à sa responsabilité.
- La connaissance est un patrimoine stratégique.
- Une erreur correctement analysée doit produire un apprentissage.
- Une réussite reproductible doit pouvoir devenir une méthode.
- La simplicité d'usage est une exigence d'architecture et non une décoration d'interface.
- Les recommandations doivent être explicables, traçables et contextualisées.
- Les valeurs de l'organisation doivent encadrer les raisonnements et les actions.
- La performance n'est pertinente que si elle est durable et transmissible.
- La Vision demeure supérieure aux choix techniques temporaires.

## 1.4 Valeurs créées

orIAna doit créer de la valeur pour :

- le client direct ;
- le client du client ;
- les collaborateurs ;
- les dirigeants et managers ;
- l'entreprise ;
- les partenaires ;
- l'écosystème ;
- les futurs utilisateurs, repreneurs ou franchisés.

La logique marketing doit notamment intégrer la lecture **client du client du client**, les vecteurs de désir, la marque commerciale et la marque employeur.

### Document(s) à enrichir

- `docs/vision/ADN_ORIANA.md` : plateforme cognitive, organisation apprenante, patrimoine cognitif, compétence collective et maturité.
- `docs/vision/CONSTITUTION_ORIANA.md` : invariants, responsabilité humaine et règles de préservation de la Vision.
- `docs/vision/CHARTE_10_PRINCIPES_ORIANA.md` : explicabilité, apprentissage et performance durable.

# 2. Cycle cognitif et organisation apprenante

## 2.1 Cycle de transformation

Le modèle cible doit représenter le cycle suivant :

> Donnée → Information → Connaissance → Compréhension → Apprentissage → Compétence → Performance → Maturité → Sagesse organisationnelle → Transmission

Chaque étape doit être définie et mesurable :

- **Donnée** : fait brut, observation ou signal.
- **Information** : donnée contextualisée et interprétable.
- **Connaissance** : information comprise, validée et réutilisable.
- **Compréhension** : capacité à expliquer les liens, causes et conséquences.
- **Apprentissage** : modification durable d'un raisonnement, d'une méthode ou d'un comportement.
- **Compétence** : aptitude démontrée à produire un résultat dans un contexte donné.
- **Performance** : création de valeur obtenue de manière mesurable.
- **Maturité** : capacité à reproduire, piloter et améliorer cette performance.
- **Sagesse organisationnelle** : capacité à arbitrer avec recul, valeurs, expérience et vision de long terme.
- **Transmission** : capacité à préserver et diffuser le savoir au-delà des individus.

## 2.2 Boucles d'apprentissage

orIAna doit organiser plusieurs boucles :

1. **Boucle individuelle** : un utilisateur reçoit une aide, agit, évalue le résultat et progresse.
2. **Boucle d'équipe** : une pratique efficace est identifiée, discutée, validée puis diffusée.
3. **Boucle organisationnelle** : les décisions, erreurs et résultats modifient les méthodes et référentiels.
4. **Boucle produit** : l'utilisation réelle révèle les améliorations nécessaires à orIAna.
5. **Boucle des Experts Virtuels** : les agents sont évalués sans apprentissage autonome non gouverné.

## 2.3 Capitalisation des erreurs et réussites

Le système doit permettre de capturer :

- le contexte initial ;
- la décision prise ;
- les hypothèses ;
- les sources mobilisées ;
- le résultat ;
- les écarts entre attendu et réel ;
- les causes ;
- l'enseignement ;
- la correction proposée ;
- la validation humaine ;
- la transformation éventuelle en règle, méthode ou connaissance.

## 2.4 Mémoire organisationnelle

La mémoire organisationnelle doit distinguer :

- mémoire documentaire ;
- mémoire décisionnelle ;
- mémoire métier ;
- mémoire client ;
- mémoire projet ;
- mémoire des erreurs ;
- mémoire des réussites ;
- mémoire des pratiques ;
- mémoire technique ;
- mémoire des agents et de leurs évaluations.

### Document(s) à enrichir

- `docs/methode/METHODE_BOREAL.md` : boucles d'apprentissage, retour d'expérience et transformation en compétence.
- `docs/methode/MOTEUR_ANALYSE_VALEUR.md` : mesure des valeurs créées et durabilité de la performance.
- Le futur contenu ORMO : cycle cognitif, types de mémoire et relations entre concepts.

# 3. Méthode BORÉAL et ORMO

## 3.1 Positionnement recommandé

BORÉAL et ORMO ne doivent pas être confondus.

- **BORÉAL** décrit la méthode : comment observer, raisonner, analyser, décider, agir, apprendre et améliorer.
- **ORMO** décrit le modèle organisé du monde d'orIAna : sur quels objets, concepts, règles, relations, valeurs et référentiels le raisonnement s'appuie.

Formulation simple :

> BORÉAL explique comment raisonner. ORMO organise ce sur quoi l'organisation raisonne.

## 3.2 Contenu attendu de BORÉAL

- observation et qualification d'une situation ;
- identification des parties prenantes ;
- compréhension des besoins explicites et implicites ;
- analyse de la valeur ;
- analyse des risques ;
- formulation d'hypothèses ;
- recherche d'éléments contradictoires ;
- proposition d'options ;
- explicitation des arbitrages ;
- décision humaine ;
- exécution ;
- mesure du résultat ;
- retour d'expérience ;
- capitalisation ;
- révision des méthodes et référentiels.

## 3.3 Contenu attendu d'ORMO

- vocabulaire métier ;

- compétences ;

- méthodes ;

## 3.4 Référentiels à préserver

# 4. Gouvernance documentaire

## 4.1 Chaîne documentaire cible

La chaîne de référence est :

> Vision → Constitution et principes → Méthode → Ontologie et référentiels → Capacités → Architecture → CDC → SPEC → PLAN → STATUS → Code → Tests → Retours d'usage → Révision contrôlée

## 4.2 Rôle des documents

- **ADN** : raison d'être, ambition et promesse.
- **Charte** : principes de conception et de comportement.
- **Constitution** : invariants, autorités, règles d'évolution et arbitrage.
- **Méthode BORÉAL** : manière de raisonner, décider et apprendre.
- **ORMO** : concepts, relations, règles et référentiels.
- **Architecture des décisions** : manière de décider et mémoriser les arbitrages.
- **CDC** : besoin global et résultats attendus.
- **SPEC** : exigences vérifiables, données, contrats et comportements.
- **PLAN** : séquencement des travaux.
- **STATUS** : état réel, décisions, écarts, blocages et reste à faire.
- **Code et tests** : réalisation vérifiable.

## 4.3 Traçabilité bidirectionnelle

Chaque capacité ou fonctionnalité devrait disposer d'identifiants stables permettant de relier :

> Principe de Vision ↔ Objectif ↔ Capacité ↔ Exigence ↔ Décision d'architecture ↔ Fonctionnalité ↔ Donnée ↔ API ↔ Écran ↔ Agent ↔ Test ↔ Code ↔ Indicateur d'usage

La traçabilité doit fonctionner dans les deux sens :

- comprendre pourquoi une fonctionnalité existe ;
- identifier quelles parties du produit réalisent un principe de Vision ;
- mesurer l'impact d'une modification ;
- détecter une capacité stratégique sans traduction fonctionnelle ;
- détecter une fonctionnalité sans justification stratégique.

## 4.4 Gouvernance des mises à jour

Toute évolution structurante doit suivre :

1. identification du besoin ou de l'idée ;
2. confrontation à la Vision et aux principes ;
3. recherche de doublons et contradictions ;
4. décision et niveau d'autorité requis ;
5. mise à jour du document de plus haut niveau concerné ;
6. propagation vers les documents aval ;
7. développement et tests ;
8. vérification documentaire après réalisation ;
9. enregistrement du retour d'expérience.

## 4.5 Dettes à suivre

- dette technique ;
- dette de sécurité ;
- dette documentaire ;
- dette fonctionnelle ;
- dette de données ;
- dette d'architecture ;
- **dette cognitive** : décisions ou pratiques non expliquées, non capitalisées ou impossibles à transmettre.

### Document(s) à enrichir

- `docs/vision/CONSTITUTION_ORIANA.md` : autorités, processus de modification et règles d'arbitrage.
- `docs/vision/ARCHITECTURE_DES_DECISIONS.md` : identifiants, statut et liens vers impacts.
- `SPEC.md` : matrice de traçabilité des exigences de la version.
- `STATUS.md` : dettes, écarts et décisions en attente.

# 5. Architecture cible

## 5.1 Source de vérité

La cible stratégique est :

- PostgreSQL comme **unique source de vérité métier** ;
- suppression complète de la dépendance métier à Grist ;
- aucune double écriture durable non gouvernée ;
- données accessibles uniquement par les services autorisés ;
- historisation et audit des modifications sensibles.

La documentation actuelle qui présente encore Grist comme source temporaire doit être mise à jour lorsque la bascule est effectivement validée. La Vision cible ne doit toutefois conserver aucune ambiguïté sur la destination finale.

## 5.2 Composants structurants

- frontend web responsive ;
- backend/API métier unique ;
- PostgreSQL ;
- stockage documentaire et médias ;
- Knowledge Center ;
- moteur de recherche lexical, structuré et sémantique ;
- AI Gateway indépendante des fournisseurs ;
- n8n ou orchestrateur de workflows ;
- Organisation Virtuelle ;
- Experts Virtuels ;
- observabilité ;
- sauvegarde et restauration ;
- gestion des identités, rôles et périmètres ;
- journalisation et audit ;
- intégrations externes.

## 5.3 AI Gateway

L'AI Gateway doit permettre :

- indépendance vis-à-vis d'un fournisseur ou modèle ;
- sélection du modèle selon le cas d'usage ;
- règles de confidentialité ;
- routage selon coût, performance, localisation et sensibilité ;
- journalisation des usages ;
- mesure des coûts par agent, utilisateur, dossier et organisation ;
- gestion des limites et quotas ;
- contrôle des prompts système ;
- évaluation des réponses ;
- protection contre l'exposition de données ;
- changement de modèle sans réécriture du métier ;
- intégration future d'une couche de protection et gouvernance telle qu'Epineon, sous réserve d'évaluation.

## 5.4 Knowledge Center

Le Knowledge Center ne doit pas être un simple espace documentaire. Il doit gérer :

- documents et versions ;
- sources ;
- métadonnées ;
- connaissances extraites ;
- décisions ;
- méthodes ;
- référentiels ;
- retours d'expérience ;
- niveaux de validation ;
- niveaux de confiance ;
- dates de validité ;
- propriétaires ;
- contradictions ;
- obsolescence ;
- droits d'accès ;
- recherche et recommandation ;
- alimentation contrôlée des Experts Virtuels.

## 5.5 Décisions d'architecture à formaliser en ADR

- PostgreSQL comme source unique de vérité.
- Sortie complète de Grist du périmètre métier.
- Backend/API comme point d'accès unique.
- Architecture multi-organisation et multi-agence.
- Stratégie d'isolation des données.
- AI Gateway et indépendance des modèles.
- Orchestration des agents et rôle de n8n.
- Organisation Virtuelle et orchestrateur principal.
- Mémoire des agents et cloisonnement des contextes.
- Knowledge Center et cycle de validation des connaissances.
- Stockage des documents, photos, sons et pièces lourdes.
- Recherche sémantique et vectorisation.
- Historisation, audit et mémoire décisionnelle.
- Explicabilité et traçabilité des recommandations.
- Observabilité et mesure des coûts IA.
- Sauvegarde, restauration et continuité d'activité.
- Sécurité by design et privacy by design.
- Stratégie d'API, connecteurs et extensibilité.
- Réversibilité des fournisseurs cloud et IA.

### Document(s) à enrichir

- `docs/vision/ARCHITECTURE_DES_DECISIONS.md` : registre et modèle des ADR.

# 11. Revue critique des orientations actuelles

## 11.1 Sortie de Grist

La conservation durable de Grist comme composant métier contredirait la cible de source unique de vérité et augmenterait la complexité. La suppression complète de Grist du métier est donc cohérente, sous réserve d'une migration contrôlée et vérifiée.

## 11.2 Nombre d'agents

Le chiffre de 23 agents ne doit pas devenir une contrainte artificielle. Il représente un inventaire historique à récupérer. La bonne unité de conception est la **capacité gouvernée**, non le nombre d'agents. Plusieurs capacités peuvent être réunies dans un Expert ; un Expert peut mobiliser plusieurs workflows.

## 11.3 Apprentissage autonome

L'apprentissage non contrôlé des agents serait contraire aux principes d'explicabilité et de responsabilité. Les corrections et résultats doivent alimenter un processus de proposition, d'évaluation et de validation humaine.

## 11.4 Multiplication documentaire

Créer un document par idée produirait une dette documentaire. Il faut enrichir les documents existants, utiliser des identifiants et ne créer de nouveaux documents que pour des objets de gouvernance réellement distincts.

## 11.5 Vision trop large

L'ambition cognitive est différenciante mais expose à la dispersion. Elle doit être préservée dans la Vision tout en étant réalisée par paliers : socle métier fiable, connaissance, assistants prioritaires, apprentissage, puis capacités avancées.

# 13. Évolutions potentiellement oubliées

## 13.1 Assistant de cohérence stratégique

Comparer en continu les nouvelles exigences et fonctionnalités à l'ADN, la Constitution, la Charte et la Méthode, puis signaler les écarts sans bloquer automatiquement.

## 13.2 Cartographie des compétences

Identifier les compétences nécessaires, détenues, démontrées, manquantes et en progression, tout en évitant une surveillance intrusive des collaborateurs.

## 13.3 Continuité du savoir

Préparer la transmission lors d'une absence, d'un changement de poste ou d'un départ : dossiers, décisions, pratiques, contacts et points de vigilance.

## 13.4 Mesure du patrimoine cognitif

Évaluer non la quantité de documents, mais la part de connaissances validées, accessibles, utilisées, actualisées et transmissibles.

## 13.5 Contradicteur virtuel

Pour les décisions importantes, mobiliser un Expert chargé de rechercher les failles, hypothèses fragiles, risques et alternatives.

## 13.6 Registre des hypothèses

Distinguer clairement faits, hypothèses, interprétations et décisions, puis vérifier les hypothèses après obtention des résultats.

## 13.7 Simulation et scénarios

Permettre de comparer des options commerciales, immobilières, organisationnelles, financières ou techniques, avec hypothèses et limites explicites.

## 13.8 Indicateur de confiance

Associer aux recommandations importantes un niveau de confiance expliqué par la qualité, la fraîcheur, la cohérence et la complétude des sources.

## 13.9 Veille multi-domaines gouvernée

Mutualiser les veilles technique, cybersécurité, IA, réglementaire, immobilière, territoriale et concurrentielle avec qualification, dédoublonnage, impacts et décisions.

## 13.10 Observatoire du produit

Analyser l'usage réel, les abandons, incompréhensions et contournements afin d'améliorer le produit sans laisser l'usage modifier silencieusement la Vision.

## 13.11 Gestion des connaissances tacites

Prévoir des entretiens guidés, récits d'expérience, démonstrations et validations par les pairs pour préserver les savoirs qui ne sont pas spontanément documentés.

## 13.12 Gouvernance économique de l'IA

Suivre le coût, la valeur, la qualité et l'empreinte de chaque usage IA ; définir des seuils et modèles de service adaptés.

# 14. Méthode de confrontation avec les autres conversations

Pour chaque idée retrouvée dans une autre conversation :

1. la reformuler fidèlement sans la simplifier excessivement ;
2. indiquer sa source et son contexte ;
3. rechercher si elle existe dans ce document ;
4. la classer comme :
   - déjà couverte ;
   - couverte partiellement ;
   - absente ;
   - contradictoire ;
   - doublon ;
   - obsolète ;
   - décision à confirmer ;
5. indiquer le chapitre concerné ;
6. proposer une modification précise ;
7. identifier le document officiel existant à enrichir ;
8. ne créer aucun document supplémentaire sans justification ;
9. distinguer Vision, capacité, fonctionnalité, composant et idée exploratoire ;
10. conserver les formulations originales importantes dans un historique de décision si elles expliquent l'intention.

# 16. Conclusion

L'ambition d'orIAna ne réside pas dans l'accumulation de fonctionnalités ou d'agents. Elle réside dans la capacité à relier métier, connaissance, décision, apprentissage, compétence et création de valeur.

Les priorités stratégiques révélées par cet audit sont :

1. harmoniser la cible PostgreSQL et la sortie de Grist dans tous les documents ;
2. formaliser BORÉAL et ORMO comme deux couches complémentaires ;
3. établir la gouvernance complète du Knowledge Center ;
4. retrouver et consolider l'inventaire historique des 23 agents ;
5. formaliser l'Organisation Virtuelle et son orchestrateur ;
6. préserver explicitement l'Assistant technique de veille ;
7. traduire l'organisation apprenante en mécanismes fonctionnels ;
8. établir la traçabilité bidirectionnelle Vision–code ;
9. gouverner les données, connaissances, coûts et résultats IA ;
10. construire par paliers sans réduire l'ambition de la Vision.

Ce document doit rester vivant, mais toute évolution doit conserver son rôle : **inventorier, confronter, révéler les manques et orienter les enrichissements des documents officiels existants**.
